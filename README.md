# gravity-build

Плагин Claude Code: собирает прототипы и приложения **«как принято»** на настоящих компонентах
[Gravity UI](https://gravity-ui.com) — без выдуманных пропов, без «похожих» компонентов, с
правильным выбором библиотеки, app-shell-обвязкой и закреплёнными версиями.

> **Бета (v0.3.0).** Knowledge-слой обкатан на ~40 прогонах и реальных приложениях; сборки проверяют
> себя сами, а проверку можно звать и отдельно (`verify`). Ставится через
> marketplace. Нашёл проблему или промах — см. [Фидбек](#фидбек).

## Что умеет

- **Идиоматичный код** — компоненты, пропы и версии сверяются с registry, а не берутся по памяти.
- **Правильная библиотека** — `@gravity-ui/uikit`, `navigation`, `table`, `page-constructor`,
  `icons`, `dynamic-forms`: выбор и обвязка под задачу (лендинг / дашборд / форма / настройки / …).
- **Сборка по Figma-макету** — дай ссылку на файл/фрейм: макет переносится в работающий код
  (компоненты и варианты — по инстансам кита, тема и раскладка — из макета, расхождения — списком
  в отчёте). Нужен доступ к Figma в среде (официальный Figma MCP).
- **Самопроверка** — после сборки прототип автоматически проверяется (контраст, картинки,
  прокрутка по ширинам, обход всех состояний экрана); скриншоты состояний остаются в `gate-out/`.
  Ту же проверку можно позвать отдельно на любой Gravity-прототип (`verify`).
- **Per-service «как принято»** — опциональный профиль накладывает тему/бренд, плотность,
  раскладку, форм-политику и словарь конкретного сервиса.

## Скиллы

| Скилл | Что делает |
|---|---|
| `create` | Собирает UI (по описанию или по Figma-макету): registry + (опц.) профиль → React + TypeScript. |
| `verify` | Проверяет любой Gravity-прототип (гейт): после сборки — с фиксами, отдельно — отчёт без правок. |
| `synthesize-profile` | Интервью → создаёт `service-profile.json` («как принято у нас»). |
| `write-back` | Сворачивает фидбек по сборке в нужный слой знания (профиль / registry / фикс). |
| `feedback` | Помогает тестеру оформить фидбек в структурированный отчёт + GitHub-issue. |

## Установка

Это **две отдельные команды** — введи первую, нажми Enter, дождись `✔`, затем вторую. Не вставляй обе разом.

Добавь marketplace:

```text
/plugin marketplace add https://github.com/ykamendrovskiy/gravity-build.git
```

Установи плагин:

```text
/plugin install gravity-build@gravity-build
```

> **Падает с `git@github.com: Permission denied (publickey)`?** Это локальная SSH-настройка, а не репозиторий — он публичный. Полный `https://…`-URL в команде выше клонирует по HTTPS (авторизация не нужна), минуя SSH.

## Использование

### Собрать по описанию

После установки просто попроси собрать что-нибудь на Gravity UI — скилл `create` подхватится сам. Например:

> Собери дашборд на Gravity UI: слева боковая навигация (Обзор, Проекты, Настройки) со сворачиванием в иконки, в центре — таблица проектов с выделением строк и группировкой по статусу.

Claude соберёт готовый к запуску проект на React + TypeScript и после зелёной компиляции сам прогонит проверку (см. [самопроверку](#самопроверка-после-сборки) ниже). Запусти проект:

```text
npm install
npm run dev
```

Так же работают лендинги, формы, страницы настроек и списки — опиши задачу, а выбор библиотеки и app-shell-обвязку плагин возьмёт на себя.

### Собрать по Figma-макету

Если дизайн уже есть в Figma — просто дай ссылку на фрейм:

> Собери прототип по этому макету: https://www.figma.com/design/…?node-id=…

Макет станет источником «что строить» (контент, раскладка, тема — включая цвета состояний контролов), а registry — «как строить» (имена, версии, обвязка). В ответе будет отчёт о расхождениях: что взято из макета, что — из справочника, где они конфликтуют. Два условия: в среде должен быть подключён официальный Figma MCP (доступ на чтение файла), а макет собран на компонентах кита Gravity. Чтение макета идёт по секциям и добавляет к цене обычной сборки примерно половину — это нормально.

Статичный макет при этом переносится живым: сетка тянется за окном, списки скроллятся, выделение и сворачивание работают — а не «картинка в браузере».

### Посмотреть все состояния прототипа

Прототип строит не только «идеальный» экран: состояния загрузки, пустого списка, ошибки, «ничего не найдено» достижимы через реальный поток данных и объявлены в `scenarios.manifest.json` в корне проекта. В dev-режиме они переключаются прямо в браузере параметром `?scenario=`:

```text
http://localhost:5173/?scenario=empty
http://localhost:5173/?scenario=submit-error
```

Список id смотри в манифесте своего проекта. В обычную прод-сборку переключатель не попадает; собрать демо-версию с живыми сценариями — `VITE_SCENARIOS=1 npm run build`.

### Панель прототипа

Вместе с состояниями сборка получает плавающую панель в углу экрана — чтобы переключать состояния мог кто угодно, без правки адресной строки. Когда она полезна:

- **Показ дизайнеру или стейкхолдеру.** Открыли прототип — и прямо на встрече листаете состояния из выпадающего списка (имена человеческие, из макета или контракта состояний: «1A — меню действий», «2 — письмо перенесено»), тут же проверяете тёмную тему переключателем.
- **Самостоятельный обход.** Ревьюер смотрит прототип без вас: панель подсказывает, какие состояния вообще есть, — ни одно не останется незамеченным.
- **Скриншоты и записи.** Панель убирается из кадра: кнопка в её шапке сворачивает панель в маленькую круглую кнопку, а `⌘+.` / `Ctrl+.` прячет целиком — как скрытие интерфейса в прототипах Figma; повторное нажатие возвращает. Подсказка про шорткат написана в самой панели.

Ещё способы управления: `?panel=0` в адресе — открыть прототип вовсе без панели (например, для встраивания); в прод-сборке панели нет так же, как и сценариев. Панель добавляется во все сборки с манифестом состояний по умолчанию — если она не нужна, скажите при сборке «без панели».

### Самопроверка после сборки

После сборки скилл сам прогоняет автоматическую проверку (гейт): контраст текста, битые картинки, горизонтальная прокрутка на ширинах от мобильной до десктопа, обход всех заявленных состояний из манифеста, известные классы ошибок по исходникам. Находки чинятся по справочнику; что не сошлось — честно описывается в ответе. Скриншоты всех состояний остаются в `gate-out/` (сводный вердикт — `gate-out/verdict.json`) — удобно посмотреть самому и показать коллегам.

Браузерным проверкам нужен Playwright (хромиум ~130 МБ — ставится только с твоего согласия); без него гейт переходит в облегчённый режим по исходникам и прямо перечисляет, какие оси не проверялись. Торопишься — скажи «собери быстро, без проверки»: шаг будет пропущен с явной пометкой в ответе.

### Проверить существующий прототип

Скилл `verify` можно звать и отдельно — на любой Gravity-прототип, не только собранный этим плагином:

> Проверь мой прототип на Gravity UI

В этом режиме гейт выдаёт отчёт и рекомендации, **не трогая код**: что найдено, почему это дефект и как чинить. Правки — только по твоей явной просьбе. Удобно как приёмка после ручных доработок.

## Обновление

Вышла новая версия — обнови marketplace и переустанови плагин:

```text
/plugin marketplace update gravity-build
```

Затем переустанови: `/plugin` → `gravity-build` → reinstall (или повтори `/plugin install gravity-build@gravity-build`).

> Важно: повторный `/plugin marketplace add …` берёт **закэшированную** копию — для обновления используй `marketplace update`, а не `add`.

## Профиль сервиса (опционально)

Чтобы плагин собирал «как принято именно у вас» — в стиле конкретного сервиса — положи `service-profile.json` в корень целевого проекта
(или `.gravity/service-profile.json`). Создать профиль через интервью — скилл `synthesize-profile`.
Без профиля сборка идёт по универсальным дефолтам.

## Фидбек

Это бета — фидбек ценен. Два пути:

- Запусти скилл **`/gravity-build:feedback`** — он соберёт структурированный отчёт и даст
  пред-заполненный issue.
- Или открой [issue вручную](https://github.com/ykamendrovskiy/gravity-build/issues/new?labels=beta-feedback).

> Репозиторий публичный — не вставляй в отчёты внутренние имена сервисов, URL и доменные данные.
