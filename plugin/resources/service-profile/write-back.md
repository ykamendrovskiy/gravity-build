# Write-back: дистилляция фидбэка в профиль (плейбук)

Как свернуть **правки с прогона** (владелец/пользователь посмотрел собранный прототип и дал фидбэк) в нужный слой знания — чтобы они соблюдались в следующих сборках «бесплатно», без повторного объяснения. Это замыкающее звено жизненного цикла: `синтез → сборка → **write-back** → сборка…`. Профиль — **живой ruleset**, нарастающий из обратной связи.

**Когда применять:** есть собранный прототип на Gravity UI + фидбэк по нему (дословные пункты). Цель — не «починить эту сборку», а извлечь **переносимое** правило и положить его в правильный слой.

**Архитектурная рамка:** дистилляция — суждение (человек/ассистент в петле), сборка по уточнённому профилю — наивный бэк. Как и интервью: интерактивный фронт → наивный бэк.

## Два слоя — куда что (граница)

Знание живёт в двух слоях, и каждый пункт фидбэка надо **развести** по ним:

- **Универсальный registry** — корректность и идиома, верные для всех сервисов (имена компонентов, версии, app-shell, «FormRow вместо инлайн-`label=`», подтверждение опасного, чекбоксы выбора ⇒ ActionsPanel).
- **Профиль сервиса** (`service-profile.json`) — вкус/раскладка/политика/бренд конкретного сервиса; **не переопределяет** корректность.

**Маркер границы:** правило про «корректно ли это вообще» → **registry**; правило про «как у нас принято среди равновалидных вариантов» → **профиль**.

## Фаза обсуждения (перед 4-way)

Перед классификацией — короткий разбор (владелец+ассистент), чтобы кодифицировать на правильной **высоте** и в правильном **месте**. Дёшево: суждение на месте + чтение корня, которое и так делаешь. Операционная выжимка — целиком ЗДЕСЬ; полный референс (рацио, глоссарий, трек обкатки) живёт в authoring-репозитории проекта и в бандл скилла НЕ входит — для работы он не нужен. Конвенция чтения: образ — мнемоника, определение — контракт (разошлись — верь определению).

**Шаг 0 — атомизируй · верифицируй · развилка домен/meta.** Одно замечание тестера часто = несколько находок (принцип → registry + вкус → profile): разложи ДО линз. Проверь само наблюдение — репорт бывает неверен (ложная атрибуция), но часто рождает *другую* реальную находку; диагноз из source, не из текста фидбэка. Развилка: находка про то, *как строить* (домен) → три линзы ниже; про *сам справочник* (спорит с собой / устарел против source / дублируется / верный факт недостижим / лежит не на том уровне) → **Meta-трек**.

**Линза 1 — Тип** (сколько обсуждения нужно): проп/API → механично (сверил `.d.ts`/source на пине — готово); размеры/раскладка → развилка «layout-идиома / апстрим-баг»; семантика/идиома → развилка «registry / profile / причуда одной сборки» (максимум обсуждения); поведение/состояния; контент/данные. Тип не эксклюзивен; показывает, где вообще нужно обсуждение.

**Линза 2 — Reach** (где живёт знание; куда идёт фикс). **Чистый вкус сервиса? Потолок = profile, reach универсального правила = 0 — пробы не нужны, Линза 2 на этом закончена; сразу к полю профиля.** Для остального: *потолок* = как высоко правило оправдано в принципе (компонент → шаред-примитив [хук/токен/миксин] → паттерн → foundation; вкус сервиса → profile). *Уровень фикса* = потолок, урезанный **blast-radius** (риск задеть чужое на общем уровне) и **пропорциональностью** воркэраунда — то есть **знание пишем на потолке, а сам фикс кладём туда, где безопасно** (часто ниже; бывает «никакой — только парк upstream», если фикс диспропорционален). **Пробы** (дёшево превращают догадку в факт): *близкая* — тот же компонент в другом контексте (проявится ли); *контрастная* — другой компонент с тем же корнем (обязательна для находок уровня «шаред-примитив»+; попадание = 2-е наблюдение бесплатно). Высокий *предсказанный* reach ≠ право сразу писать широко — сперва пробей контрастный кейс.

**Линза 3 — Высота** (как сформулировать; для registry-находок): веди **интентом**, не механизмом/воркэраундом (баг → в скобку-«почему»; правило должно пережить апстрим-фикс) · registry vs profile · контракт, не инстанс · полнота для наивного агента (API + раскладка). **Кодифицируешь воркэраунд/версионно-гейтнутый факт → пометь WA-тегом** (ADR-0008: `<!-- WA <pkg>@<ver> · recheck: <что перепроверить на бампе> · upstream: <статус> -->` — внутренний, из бандла стрипается) — иначе на бампе Гравити он молча сгниёт.

**Meta-трек** (находка про сам справочник, не про домен): доменные линзы не применяются — чини **структуру/консистентность** (не «дописать знание»), grounded в pointer-model «факт → один guide; без дублей». Kind → действие: *противоречие / дрейф против source / дублирование* → сверить source, привести к канону / схлопнуть в один; *reachability* (факт есть, но не на пути чтения) → перенести на ходовой путь — частный случай: **recipe — не дом для уникальных фактов** (recipe = композиция; API/library-факт только в recipe → в каталог, recipe ссылается — ADR-0001-доп.); *размещение* (вкус как универсальное / правило ниже потолка) → переместить по reach. Реактивная meta-находка из write-back'а и проактивный аудит «ревизия закодифицированного» — один мускул (здоровье registry).

## 4 типа пункта (классифицируй каждый)

1. **Write-back в профиль** — пункт выражает конвенцию сервиса (новую или уточнение): эстетика, раскладка, политика, словарь. → Конкретная дельта: какое **поле** профиля (`theme` / `density` / `contentLayout` / `formPolicy` / `navigation` / `interactionPolicy` / `feedbackPolicy` / `componentStyling` / `approvedComponents` / `archetypes` / `glossary`) или строка `conventions` (escape-hatch для ещё-не-структурированного).
2. **Кандидат в registry** — пункт про универсальную корректность/качество, верную для любого сервиса. → Куда (какой item: `library-uikit` / `recipe-*` / `pattern-app-shell` / `pattern-theming`) и какое **переносимое** правило. registry автоматически **не правим** — выдаём отчёт-кандидат мейнтейнеру.
3. **Дефект сборки** — конвенция/корректность уже описаны, но реализованы с багом. → Профиль и registry не трогаем (или трогаем отдельно — обоснуй), нужен фикс кода.
4. **Подтверждение** — пункт подтверждает уже записанную конвенцию: изменений не требуется.

Один пункт может породить **больше одного** действия (напр. дефект сборки + кандидат-правило в registry, чтобы класс бага не повторялся).

## Правила дистилляции

- **Аномалия ≠ конвенция** (канон-формула — `dimensions.md` «Правила синтеза»): противоречащее универсальной идиоме — **флаг-вопрос владельцу**, не молчаливая запись в профиль.
- **Невоспроизведённый дефект — не сочиняй механизм.** Если баг не воспроизведён эмпирически — помечай «нужна репродукция», а не достраивай правдоподобное объяснение.
- **Правила переносимы.** Формулируй как класс ошибок («фикс-размерный элемент во flex-ряду → `flex-shrink:0`»), не «в этой сборке поправить вот тут».
- **Омиссия.** Не выдумывай конвенций сверх сказанного. Не хватает данных для уверенной классификации → задай уточняющий вопрос, а не угадывай.
- **Курирование.** Различай политику сервиса и причуду одной сборки; в профиль — только устойчивое.
- **Дельта минимальна и схема-валидна.** Меняй ровно то, что следует из фидбэка; профиль остаётся валидным по схеме.

## Поток

1. **Вход:** профиль (`service-profile.json`), код сборки (только чтение), фидбэк (дословные пункты), registry-снапшот (для классификации в его сторону).
2. **По каждому пункту:** сперва **фаза обсуждения** (Шаг 0 → домен: 3 линзы / meta: Meta-трек — секция выше), затем классификация (1–4 или комбинация) + обоснование в 1–2 строки + конкретное действие.
3. **Применение:**
   - профильные пункты → дельта в `service-profile.json` (структурное поле или `conventions`), минимально, схема-валидно;
   - registry-кандидаты → **отчёт** (item + переносимое правило), без авто-правки registry;
   - дефекты → описание фикса; невоспроизведённые → пометка «нужна репродукция»;
   - аномалии/неясности → вопрос владельцу.
4. **Выход:** обновлённый профиль (дельта) + отчёт (registry-кандидаты / дефекты / вопросы). По возможности обнови `provenance` профиля (что добавлено write-back'ом и из какого фидбэка).
