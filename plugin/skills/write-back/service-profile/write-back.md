# Write-back: дистилляция фидбэка в профиль (плейбук)

Как свернуть **правки с прогона** (владелец/пользователь посмотрел собранный прототип и дал фидбэк) в нужный слой знания — чтобы они соблюдались в следующих сборках «бесплатно», без повторного объяснения. Это замыкающее звено жизненного цикла: `синтез → сборка → **write-back** → сборка…`. Профиль — **живой ruleset**, нарастающий из обратной связи.

**Когда применять:** есть собранный прототип на Gravity UI + фидбэк по нему (дословные пункты). Цель — не «починить эту сборку», а извлечь **переносимое** правило и положить его в правильный слой.

**Архитектурная рамка:** дистилляция — суждение (человек/ассистент в петле), сборка по уточнённому профилю — наивный бэк. Как и интервью: интерактивный фронт → наивный бэк.

## Два слоя — куда что (граница)

Знание живёт в двух слоях, и каждый пункт фидбэка надо **развести** по ним:

- **Универсальный registry** — корректность и идиома, верные для всех сервисов (имена компонентов, версии, app-shell, «FormRow вместо инлайн-`label=`», подтверждение опасного, чекбоксы выбора ⇒ ActionsPanel).
- **Профиль сервиса** (`service-profile.json`) — вкус/раскладка/политика/бренд конкретного сервиса; **не переопределяет** корректность.

**Маркер границы:** правило про «корректно ли это вообще» → **registry**; правило про «как у нас принято среди равновалидных вариантов» → **профиль**.

## 4 типа пункта (классифицируй каждый)

1. **Write-back в профиль** — пункт выражает конвенцию сервиса (новую или уточнение): эстетика, раскладка, политика, словарь. → Конкретная дельта: какое **поле** профиля (`theme` / `density` / `contentLayout` / `formPolicy` / `navigation` / `interactionPolicy` / `feedbackPolicy` / `componentStyling` / `approvedComponents` / `archetypes` / `glossary`) или строка `conventions` (escape-hatch для ещё-не-структурированного).
2. **Кандидат в registry** — пункт про универсальную корректность/качество, верную для любого сервиса. → Куда (какой item: `library-uikit` / `recipe-*` / `pattern-app-shell` / `pattern-theming`) и какое **переносимое** правило. registry автоматически **не правим** — выдаём отчёт-кандидат мейнтейнеру.
3. **Дефект сборки** — конвенция/корректность уже описаны, но реализованы с багом. → Профиль и registry не трогаем (или трогаем отдельно — обоснуй), нужен фикс кода.
4. **Подтверждение** — пункт подтверждает уже записанную конвенцию: изменений не требуется.

Один пункт может породить **больше одного** действия (напр. дефект сборки + кандидат-правило в registry, чтобы класс бага не повторялся).

## Правила дистилляции

- **Аномалия ≠ конвенция.** Наблюдение, противоречащее универсальной идиоме, — **флаг-вопрос владельцу**, не молчаливая запись в профиль (может быть причудой одной сборки — «случайные конвенции»).
- **Невоспроизведённый дефект — не сочиняй механизм.** Если баг не воспроизведён эмпирически — помечай «нужна репродукция», а не достраивай правдоподобное объяснение.
- **Правила переносимы.** Формулируй как класс ошибок («фикс-размерный элемент во flex-ряду → `flex-shrink:0`»), не «в этой сборке поправить вот тут».
- **Омиссия.** Не выдумывай конвенций сверх сказанного. Не хватает данных для уверенной классификации → задай уточняющий вопрос, а не угадывай.
- **Курирование.** Различай политику сервиса и причуду одной сборки; в профиль — только устойчивое.
- **Дельта минимальна и схема-валидна.** Меняй ровно то, что следует из фидбэка; профиль остаётся валидным по схеме.

## Поток

1. **Вход:** профиль (`service-profile.json`), код сборки (только чтение), фидбэк (дословные пункты), registry-снапшот (для классификации в его сторону).
2. **По каждому пункту:** классификация (1–4 или комбинация) + обоснование в 1–2 строки + конкретное действие.
3. **Применение:**
   - профильные пункты → дельта в `service-profile.json` (структурное поле или `conventions`), минимально, схема-валидно;
   - registry-кандидаты → **отчёт** (item + переносимое правило), без авто-правки registry;
   - дефекты → описание фикса; невоспроизведённые → пометка «нужна репродукция»;
   - аномалии/неясности → вопрос владельцу.
4. **Выход:** обновлённый профиль (дельта) + отчёт (registry-кандидаты / дефекты / вопросы). По возможности обнови `provenance` профиля (что добавлено write-back'ом и из какого фидбэка).
