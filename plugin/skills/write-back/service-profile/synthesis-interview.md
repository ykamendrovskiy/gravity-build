# Авто-синтез профиля: интервью-режим (плейбук)

Как собрать `service-profile` через **диалог** (а не extraction из кода). Это **интерактивный фронт** авто-синтеза: интервьюер ведёт по `dimensions.md`, но не анкетой — выводит из описания, спрашивает только дыры, остальное оставляет на дефолте. Пара: `dimensions.md` = *что* распознавать (контент), этот файл = *как* вести (поток).

**Когда применять:** новый сервис / кода ещё нет / либо как фаза 2 поверх extraction (дозадать то, что детект из кода оставил неопределённым).

**Архитектурная рамка:** интервью **интерактивно** (человек в петле) → ведёт его интерактивный ассистент, а не headless-агент. Сборку из готового профиля делает наивный/headless-агент (как прогоны 01/02). Итог: *интерактивный фронт (интервью) → наивный бэк (сборка).* «Авто» здесь = ассистированное, не автономное (рифмуется с HONK).

## Принципы (что сработало в триале)

1. **Открывай описанием сервиса — оно высокодоходно.** Характер («плотно, видно много сразу» / «запас воздуха, легковесно») напрямую задаёт density, contentLayout, approvedComponents, archetypes, glossary. В триале ~6 из 8 измерений вышли из 1–2 абзацев.
2. **Вывести-потом-подтвердить, не допрашивать.** По каждому измерению сперва попробуй вывести из описания; **спрашивай только если** оно варьируется и не выводимо.
3. **Омиссия = дефолт.** Не спрашивай про всё. Неотвеченное → универсальный дефолт. Не выдумывай (tempo без бренда → дефолтный акцент; forge `requiredMarking` не спрашивали → дефолт per-field).
4. **Вопрос — с дефолтом как явной опцией.** Чтобы пользователь мог быстро принять дефолт, а не сочинять.
5. **«Заявлено, но ещё нет» — изящно.** Напр. бренд назван, а Themer-темы пока нет → `themeConfig` опусти + пометь TODO в `note`. Профиль остаётся валидным и рабочим, достроится позже.
6. **Фиксируй провенанс.** `provenance.synthesizedFrom` = «Интервью + какие вопросы задавались»; в `note` помечай, что выведено / спрошено / оставлено на дефолте.

## Поток

1. **Описание сервиса** — один открытый вопрос: что это и домен; кто пользователь и в каком контексте (внутренний инструмент / публичный продукт / админка); характер интерфейса (плотный-быстрый / просторный-«лицо продукта» / строгий).
2. **Вывод по `dimensions.md`** — для каждого измерения примени логику ниже к **тексту описания** (аналог колонки «как обнаружить», но по словам, не по коду): что описание уже задало.
3. **Точечные вопросы по дырам** — только измерения, что описание не закрыло и которые важны; дефолт — опцией.
4. **Сборка** — `service-<slug>.json` + валидация по схеме (`uv run --with jsonschema …`).
5. **Презентация** — что вывел / что спросил / что на дефолте; провенанс.

## Карта «вывести vs спросить» по измерениям

| Измерение | Что обычно задаёт описание (→ вывести) | Спросить, если |
|---|---|---|
| **theme** | посыл («технологичный») ≠ light/dark — выводится **редко** | **почти всегда:** light/dark, переключатель, есть ли бренд (→ Themer-тема или TODO) |
| **density** | «плотно / много сразу» → compact; «воздух / легковесно» → spacious | характер плотности не назван |
| **contentLayout** | «много сразу / большой монитор» → full-width; «публичный / воздух» → центр+max-width; платформа (Electron/web/on-prem) → `adaptive` | непонятны плотность или платформа |
| **formPolicy** | при **spacious public** надёжно: сверху + FieldDescription + uniform; при **compact** — варьируется | плотный инструмент (компактная форма неоднозначна: слева+HelpMark+by-content vs просторнее) |
| **approvedComponents** | домен → набор (таск-трекер → `@gravity-ui/table`; лендинг/описание → `page-constructor`) | есть явные политики «нельзя / только» (allow/denylist) |
| **archetypes** | домен → типовые экраны (список+карточка; настройки+промо) | нестандартные экраны |
| **glossary** | домен → термины | спец. словарь / брендовые формулировки |
| **referenceScreens** | — (кода нет, в интервью не появляется) | команда сама укажет эталонные экраны |
| **navigation** | домен/характер намекает (внутр. инструмент → AsideHeader+крошки; публичный → footer, меньше крошек) | **почти всегда уточнить:** есть ли главная; политика и размещение крошек (отдельно/в ActionBar); футер страницы |
| **interactionPolicy** | редко выводимо | спросить про **unsaved-гард** и save-модель (особенно где есть формы редактирования) |
| **feedbackPolicy** | характер намекает на глубину (ops/24-7 → rich; публичный → polished; внутр. → standard) | уточнить глубину, если важна надёжность (индикаторы активности, async-retry) |
| **componentStyling** | редко выводимо | спросить: иконки на кнопках или только текст; политика `view` |

Эмпирика триала (`service-tempo` / `service-forge`): спрашивать пришлось **тему** (оба) + **форм-политику** (forge, т.к. compact) — остальное вывелось.

## Связь с другими режимами

- **Extraction** (когда есть код/макеты): детект по `dimensions.md` вместо вопросов; интервью тогда — фаза 2, дозадать только дыры extraction.
- **Write-back-луп:** правки с прогонов (фидбэк → правила) дописываются в `conventions`, периодически структурируясь — профиль как живой ruleset.

## Для наивного прогона (симуляция интервью без человека)

Чтобы протестировать **сам плейбук** наивными агентами (а не привилегированным ассистентом): **агент-владелец** (бриф сервиса + отвечает «в характере») ↔ **агент-интервьюер** (этот файл + `dimensions.md` + схема, ведёт диалог) → транскрипт → профиль; затем **наивный сборщик** + registry + профиль → экран → Tier 0/1. Headless-агент не говорит с живым человеком, поэтому интервью здесь **симулируется** между агентами — это офлайн-валидация механики, не продовый поток.
