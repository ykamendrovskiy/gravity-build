# Write-back — свернуть фидбэк прогона в профиль

Возьми **правки по собранному прототипу** (фидбэк владельца/пользователя) и разведи их по слоям
знания, чтобы они соблюдались в следующих сборках «бесплатно». Профиль — живой ruleset.

## Когда применять

- Есть собранный прототип на Gravity UI **+ фидбэк** по нему, и нужно свернуть правки так, чтобы они
  переносились на следующие сборки (а не правились вручную каждый раз).

## Как вести

1. **Прочитай плейбук и контекст** (общие ресурсы плагина, пути — от папки этого скилла):
   `../../resources/service-profile/write-back.md` (как дистиллировать: 4 типа пункта + правила) +
   `../../resources/service-profile/dimensions.md` (поля профиля) +
   `../../resources/service-profile/structural-conventions.md` (граница структурно-поведенческих правил) +
   `../../resources/service-profile/schema/service-profile.schema.json` (валидность дельты).
   Для классификации **в сторону registry** — `../../resources/registry/registry.json` +
   `../../resources/registry/guides/*` (структура универсального слоя).
2. **Вход:** существующий `service-profile.json` (корень целевого проекта), код сборки (только
   чтение), фидбэк (дословные пункты).
3. **По каждому пункту — классифицируй** по границе (4 типа): **write-back в профиль** (вкус/политика)
   / **кандидат в registry** (универсальная корректность) / **дефект сборки** (баг реализации) /
   **подтверждение**. Один пункт может дать больше одного действия. Применяй правила плейбука:
   аномалия ≠ конвенция (флаг-вопрос); невоспроизведённый дефект → «нужна репродукция» (не сочиняй
   механизм); правила переносимы (класс ошибок); курирование против «случайных конвенций».
4. **Применение:**
   - профильные пункты → минимальная схема-валидная **дельта в `service-profile.json`** (структурное
     поле или `conventions`); обнови `provenance`;
   - registry-кандидаты → **отчёт** (item + переносимое правило); registry автоматически **не правь**;
   - дефекты → описание фикса; аномалии/неясности → вопрос владельцу.
5. **Выход:** обновлённый профиль (дельта) + отчёт (registry-кандидаты / дефекты / вопросы).

## Связь

- Профиль создаёт `synthesize-profile`; собирает по нему скилл сборки (`create`).
  Write-back замыкает цикл «синтез → сборка → уточнение».
