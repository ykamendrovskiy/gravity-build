# Шаблон отчёта фидбека

Заполни и выдай тестеру этот markdown. Форма совпадает с входом скилла `write-back` (per-item:
дословный пункт + severity + категория-подсказка + repro), чтобы классификация владельца была
механической.

```markdown
## gravity-build · beta feedback

**Сборка:** <что собирал, дословно со слов тестера>
**Скилл(ы):** gravity-build | synthesize-profile | write-back | feedback
**Профиль:** есть (<сервис/характер>) | нет (baseline)

**Окружение** (снято автоматически, где возможно):
- plugin: gravity-build @ <commit | version>
- gravity: uikit <v>, icons <v>, navigation <v>, page-constructor <v>, … (из package.json проекта)
- node / OS: <… если релевантно>
- код: <путь к сгенерированному прототипу>

### Пункты
1. **[blocker | major | minor | nit]** <дословно: что не так>
   - категория (подсказка тестера, НЕ диагноз): wrong-component | wrong-prop | version-mismatch | build-error | runtime-error | profile-misapply | idiom/aesthetic-miss | suggestion
   - ожидал: <…> / получил: <…>
   - repro: <минимальные шаги ИЛИ сниппет + файл:строка> | «не воспроизвёл — нужна репродукция»
2. …
```

## Правила заполнения

- **Один пункт = одна проблема.** Не склеивай разные промахи в один буллет.
- **Дословность.** Формулировку проблемы бери словами тестера, не переписывай в свою интерпретацию.
- **Severity:** blocker (не собралось / не запускается) · major (собралось, но грубо не так) ·
  minor (заметный недочёт) · nit (мелочь/вкусовщина).
- **Категория — подсказка, не вердикт.** Тестер не знает границы registry/профиль; владелец
  переклассифицирует во `write-back`. Если не уверен — оставь без категории.
- **Repro обязателен по смыслу:** либо шаги/сниппет, либо честное «не воспроизвёл». Никаких
  выдуманных механизмов.
- **Окружение — авто.** Тяни версии из `package.json`/`plugin.json`, не спрашивай тестера, если
  можешь прочитать сам.
