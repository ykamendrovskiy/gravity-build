# Feedback — сообщить о проблеме со сборкой

Помоги **тестеру беты** чисто и структурированно описать фидбек по прототипу, собранному плагином
`gravity-build`, и довести его до владельца через GitHub-issue. Отчёт строится в форме, которую
напрямую потребляет скилл владельца `write-back` (классификация по слоям знания).

**Захват, не диагноз.** Ты фиксируешь фидбек как есть; **не** раскладываешь его по слоям
(профиль / registry / дефект) — это суждение владельца во `write-back`. Можешь приложить подсказку
категории со слов тестера, но помечай её как неавторитетную.

## Когда применять

- Тестер собрал (или пытался собрать) что-то плагином `gravity-build` **и хочет сообщить**: промах
  по компоненту/пропу/версии, ошибка сборки или рантайма, профиль применился не так, эстетический /
  идиоматический промах, либо предложение.

## Как вести

1. **Прочитай шаблон** `report-template.md` (рядом с этим `SKILL.md`) — это форма отчёта. Заполняй
   строго по ней.
2. **Сними окружение автоматически**, насколько можешь, не допрашивая тестера:
   - версия плагина — из `repository`/`version` в его `plugin.json` (или git-commit установки);
   - версии `@gravity-ui/*` — из `package.json` целевого проекта;
   - путь к сгенерированному коду; node/OS — если релевантно дефекту.
3. **Собери пункты фидбека** — по одному на проблему, **дословно** со слов тестера. Для каждого:
   severity (blocker/major/minor/nit); подсказка категории (со слов тестера, не диагноз); ожидал vs
   получил; **repro** — минимальные шаги или сниппет+файл.
   - **Дисциплина repro:** если воспроизведения нет — пиши «не воспроизвёл — нужна репродукция».
     **Не сочиняй механизм** дефекта, которого не видел.
4. **Покажи тестеру заполненный отчёт** (markdown по шаблону) на проверку.
5. **Выдай пред-заполненный GitHub-issue:**
   `https://github.com/ykamendrovskiy/gravity-build/issues/new?title=<URL-энкод>&body=<URL-энкод>&labels=beta-feedback`
   — `title` краткий из сути, `body` = отчёт. URL-энкодь title и body.
   - **Лимит URL (~8 КБ):** если отчёт длинный — выдай его как markdown-блок и попроси тестера
     вставить в issue вручную (issue открой по короткому URL без `body`).
   - **Альтернатива:** если у тестера есть `gh` — предложи `gh issue create`. Если постить публично
     нельзя (внутренний тестер) — выдай только markdown для приватной передачи владельцу.

## Безопасность (репозиторий публичный)

Issue уходит в **публичный** репозиторий. **Перед формированием отчёта предупреди тестера:** «отчёт
станет публичным — убери внутренние имена сервисов, URL, доменные данные». Не включай ничего, что
тестер не готов опубликовать.

## Чего не делать

- Не классифицировать авторитетно по слоям (профиль/registry/дефект) — это владелец во `write-back`.
- Не отправлять issue за тестера — он смотрит отчёт и сабмитит сам.
- Не выдумывать repro/механизм невоспроизведённого дефекта.

## Связь

- Отчёт этого скилла = вход скилла владельца `write-back` (он классифицирует и сворачивает фидбек в
  профиль / кандидат в registry / фикс). Так замыкается петля «сборка → фидбек тестера → учёт».
