# gravity-build: verify — проверка сборки гейтом

Headless-проверка Gravity-прототипа по инвариантам качества + обход всех заявленных состояний.
Операционная механика (команды, режимы полный/облегчённый, чтение вердикта) — **`harness/HARNESS.md`**
(рядом с этим SKILL.md); там же дисциплина «чини классами по гайдам, максимум 2 итерации».

## Два режима — определи по контексту вызова

- **«После сборки»** (шаг самопроверки из скилла `create`, либо пользователь просит «собери и проверь»):
  находки гейта **чини сам** — классами по гайдам registry (общий ресурс плагина:
  `../../resources/registry/guides/`), затем re-gate;
  что не сошлось за 2 итерации — честно в отчёт.
- **«Standalone-аудит»** (пользователь навёл на СУЩЕСТВУЮЩИЙ проект: «проверь мой прототип»):
  **по умолчанию — отчёт, НЕ фиксы** (это чужой код). Вердикт → человекочитаемый разбор: что найдено,
  почему это дефект (ссылкой на канон registry), как чинить — рекомендацией. Правки — только по явной
  просьбе пользователя.

## Порядок работы

1. Определи корень проверяемого проекта (package.json). Есть `scenarios.manifest.json` → гейт обойдёт
   все состояния; нет — одиночный аудит начального экрана (тоже валидно).
2. Сборка для гейта: `VITE_SCENARIOS=1 npm run build` (переключатель состояний должен быть живым;
   без него гейт честно скажет «сценарии noop»). Предпросмотр — на ВРЕМЕННОМ свободном порту.
3. `node <этот-скилл>/harness/run-gate.mjs http://localhost:<порт> <проект> <проект>/gate-out` —
   дальше по `harness/HARNESS.md` (полный режим при наличии Playwright; иначе честный static-only).
4. Разбери `verdict.json` по режиму (см. выше). Скриншоты `gate-out/full-*.png` — посмотри сам
   (это «глаза» на всех состояниях) и предложи пользователю.
5. Отчёт: что проверено и НАСКОЛЬКО полно (полный/частичный режим — явно), находки/рекомендации,
   пути к вердикту и скриншотам.

## Чего не делать

- Не редактировать файлы `harness/` и не подгонять проверяемый код «под чек» — находки чинятся по
  гайдам; подозрение на ошибку гейта — опиши, не «чини» здоровый код.
- Static-only-вердикт не выдавать за полную проверку (browser-линии в нём пропущены).
- В standalone-аудите не менять чужой код без явной просьбы.
- Не занимать порт, который пользователь просил для своего предпросмотра, — гейту хватает временного.
