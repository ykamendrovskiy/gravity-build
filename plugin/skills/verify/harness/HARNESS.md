# Самопроверка сборки (headless-гейт)

Автоматическая проверка собранного прототипа: контраст текста (APCA — в светлой теме везде, плюс
тёмный проход на дефолт-состоянии свапом темы), битые картинки, пустые image-слоты, разъезд размеров
контролов в ряду, `[object Object]` в текстах, горизонтальный оверфлоу по сетке ширин
(десктоп→мобильная), обход всех состояний из `scenarios.manifest.json`, плюс static-паттерны
известных ошибок по исходникам.

## Запуск

```bash
# 1) сборка с живыми сценариями (обычный build вырезает dev-переключатель состояний)
VITE_SCENARIOS=1 npm run build
# 2) предпросмотр для гейта — на ВРЕМЕННОМ свободном порту (любом; не занимай порт,
#    который пользователь просил для финального показа)
npx vite preview --port <временный> --strictPort &
# 3) гейт (из папки скилла; <проект> = корень собранного проекта)
node <путь-к-скиллу>/harness/run-gate.mjs http://localhost:<временный> <проект> <проект>/gate-out
# опционально: --fail-on=any|high → exit 1 при находках (для скриптовых/CI-обвязок; без флага exit 0,
#   вердикт интерпретируется по verdict.json)
# 4) после зелёного гейта временный preview можно погасить
```

Результат: `<проект>/gate-out/verdict.json` + скриншоты `full-*.png` по каждому состоянию + сводка в консоли.
Гейт сам прогревает сервер перед аудитом (первый рендер свежеподнятого preview бывает «сырой»).

**Финальный показ пользователю — отдельно от гейта:** удобнее оставить обычный `npm run dev` на порту,
который просил пользователь (в dev сценарии `?scenario=` живые сами по себе + работает HMR); гейтовый
flag-build при этом остаётся честным прод-билдом. Скриншоты гейта (`full-*.png`) полезно просмотреть и
самому — это твои «глаза» на всех состояниях сразу.

## Полный режим и деградация

- **Полный режим** требует Playwright в проекте: `npm i -D playwright && npx playwright install chromium`
  (хромиум ~130 МБ — спроси пользователя перед установкой).
- **Без Playwright** гейт сам переключается в **static-only**: проверяются только паттерны по исходникам,
  вердикт помечается `mode: "static-only"` и «browser-линии пропущены — вердикт ЧАСТИЧНЫЙ».
  Это честный частичный результат, а НЕ «всё чисто».

## Как читать вердикт и что чинить

- `summary` — счётчики по инвариантам; всё по нулям и `scenario_noop: []` → сборка чиста.
- `contrast_dark_*` + `dark_pass` — контраст тёмной темы (дефолт-состояние): `done` = мерили обе темы;
  `skipped_no_g_root` = свапнуть тему не удалось, тёмный контраст НЕ проверен (не «чисто»).
- Находки чини **классами по гайдам registry** (у каждого инварианта есть свой guide-канон: контраст —
  `library-page-constructor`/`gravity-foundations-color`; оверфлоу — reset/скролл-модели; размеры в ряду —
  `library-uikit`; пустые слоты — `library-illustrations`), а не точечными заплатками под чек.
- `scenario_noop` — задекларированный в манифесте сценарий не меняет UI: либо пресеть состояние
  (идиома `FILLED_PRESET` в `scaffold-app-shell`), либо убери id из манифеста.
- `fatal: page_unreachable_or_blank` — сервер не отвечает или страница пустая: вердикт НЕ валиден,
  проверь preview.
- Максимум **2 итерации** фикс→повторный гейт на одну находку; не сходится — честно опиши остаток
  пользователю (не подгоняй код под чек и не правь файлы харнеса).

## Чего гейт НЕ проверяет (зелёный гейт ≠ полная проверка)

- **Доступность за пределами контраста:** клавиатурная навигация/фокус, скрин-ридеры, aria-семантика.
- **Порог контраста — «ловим сломанное», не «доступно»:** |Lc|<15 = невидимый, <30 = плохой; для
  комфортного body-текста APCA рекомендует ~Lc 60+ — зелёный контраст-гейт не сертификат доступности.
- **Фон под текстом** резолвится по цветам слоёв (`backgroundColor` + `::before`); градиент/фото-фон
  гейт не видит — текст на изображениях проверяй глазами (скриншоты `full-*.png`).
- **Контент и смысл:** правдоподобность данных, i18n-длины строк, RTL, копирайт.
- **Интеракции сверх сценариев манифеста:** drag-and-drop, клавиатурные шорткаты, реальная сеть.
Это осознанная граница: находки этих классов — владельческий просмотр/vision, не headless-инварианты.
