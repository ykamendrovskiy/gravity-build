# Registry — гид для агента

Knowledge-слой для сборки прототипов на настоящих компонентах Gravity UI. Используй его, чтобы взять правильную библиотеку и собрать «как принято», а не по памяти.

## Порядок работы

1. **Сначала — метаструктура.** Если это страница или приложение из нескольких экранов: набросай структуру (какие экраны / секции, какие данные в каждой, какие взаимодействия) и только потом подбирай под каждый блок библиотеки. Идёшь от характера задачи: «рассказать о продукте» → маркетинговая страница; «постоянно работать с большим объёмом данных» → дашборд или таблица; «ввести / изменить параметры» → форма. Для одиночной простой задачи (одна форма, один список) — сразу к шагу 2. **Строишь целый composite-интерфейс** (почта, дашборд, таск-трекер, каталог) — сперва загляни в `r/reference-interfaces.json`: там эталонные реализации на Гравити (изучи структуру, не копируй вслепую).
2. **Роутинг библиотек — `r/library-routing.json`.** Сопоставь каждый блок метаструктуры со строками таблицы (`want` → `use`, `avoid`, `why`) и возьми указанную библиотеку.
3. **Открой recipe задачи.** По `triggers` найди подходящий `r/recipe-*.json`: там пакеты, routing-строки, анти-паттерны и ссылка на гайд (`doc`).
4. **Положи обвязку.** `r/pattern-app-shell.json` + `recipes/app-shell.md` — обязательный первый слой (ThemeProvider + стили, опц. Toaster / Mobile).
5. **Закрепи версии.** Возьми версии пакетов из `r/version-index.json`, не выдумывай. Если сроутился на route-only либу (markdown-editor, даты) — ставь её **bundle** целиком (пакеты + peer-deps вместе), иначе ломается install-граф.

## Принципы

- Сверяй каждое имя компонента и пропа с registry — не подставляй из памяти других дизайн-систем. Для частых компонентов uikit имена сверяй по `r/library-uikit.json` (инвентарь экспортов uikit@7) **перед** импортом — ловит stale-имена (напр. `RadioButton` → `SegmentedRadioGroup`).
- Когда слово неоднозначно (напр. «table» — в Гравити их три), роутингуй по нужной фиче, а не наугад.
- Нужна детальная API/конфиг выбранной библиотеки (точные пропы, форма sub-блоков) — `r/library-docs.json`: если у библиотеки есть свои upstream AI-доки (page-constructor / uikit / charts…), **открой их и не угадывай**; если апстрим молчит (navigation / table / icons…) — используй наш recipe/brief.
- **Если прямой фетч upstream заблокирован** (WebFetch / gh недоступны) — попробуй **WebSearch** по README/AGENTS библиотеки (так находится реальный API). И только если и это не вышло — собери из проверенных registry-примитивов и **пометь комментарием**; НЕ хардкодь компонент/пропы по памяти от безысходности (даже если кажется, что компонент такой есть — напр. `uikit/Stepper` существует, но его API надо проверить, а не сочинить).

## Профиль сервиса (если передан)

Если вместе с registry тебе дали **профиль сервиса** (`service-profile`, отдельный JSON — не часть этого registry), он задаёт «как принято именно в этом сервисе»: тема/бренд, плотность, раскладка контента, политика форм, разрешённый сабсет компонентов, архетипы, доменный словарь. Применяй его так:

- **Профиль уточняет открытые ручки**, которые гайды оставили на дефолте (плотность, выравнивание/max-width контента, ширина полей формы и размещение описаний, тема). По каждой такой ручке бери значение из профиля вместо дефолта гайда.
- **Профиль не отменяет корректность.** Имена компонентов, версии, app-shell-обвязка, «FormRow вместо инлайн-`label=`» — это registry, профиль их не переопределяет. Если директива профиля противоречит корректности — следуй registry и пометь это комментарием.
- **Чего профиль не задал — бери из дефолта** гайда (омиссия = дефолт). Профиль может быть тонким.
- `approvedComponents` сужает выбор: `deny` — не использовать, `preferred` — разрешает универсальные «it depends»-развилки (напр. «точное число → NumberInput»).
- **Тема/бренд:** если в профиле `theme.themeConfig` — следуй `pattern-theming`: скопируй указанный `.theme.css` в проект и импортируй его **после** uikit `styles.css`, токены руками не подбирай. Если `theme.brandTokens` — полный набор per-theme. Если ничего — дефолтная тема Гравити.

Если профиль **не передан** — игнорируй этот раздел и собирай по универсальным дефолтам.

## Типы item'ов

- `routing` — карта «намерение → библиотека» (entry-point).
- `recipe` — гайд под тип задачи (triggers, пакеты, routing, анти-паттерны, doc).
- `pattern` — сквозная обвязка (app-shell).
- `version-index` — закреплённые версии.

Полный индекс item'ов — `registry.json`.
