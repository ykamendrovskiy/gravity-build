# pattern-form — форма / страница настроек (агностичный спек вида)

> Тип: `pattern`. Что есть хорошая форма (параметры, настройки, диалог добавления), безотносительно сборки.
> Композиция + раскладка + контракт. **Ссылается** на `interface-foundations` (feedback, data-safety).
> Гравити-сборка (код, контролы, app-shell) — `recipe-settings-form`.

## Композиция

1. Поля, сгруппированные по смыслу (группы с подзаголовками при >~6 полях).
2. Подпись + (опц.) описание у каждого поля.
3. Валидация — per-field, видимая.
4. Действия: одно основное (Сохранить) + вторичное (Отмена).

## Раскладка (что делает форму хорошей)

- **Подпись + описание поля → строка-обёртка (`FormRow`), а не инлайн `label=` контрола.** Обёртка даёт сетку
  «подпись ↔ контрол» и слот описания; инлайновый `label=` запихивает подпись ВНУТРЬ контрола (это для
  компактного инлайна, не для страницы настроек). `FormRow` — один из вариантов (vs ручная обёртка лейбл+контрол);
  выбирай обёртку, не инлайн.
- **Ширина контролов выровнена**, а не «каждый по своему контенту»: в секции — единая ширина по большему из
  значимых полей (правые края выровнены). Узкие по смыслу (процент, счётчик) можно оставить узкими, но не рви
  колонку. Ориентир: число / код / процент — узко (≈120–200px); название / slug — средне (≈280–360px); описание — широко.
  **«Узко» задавай `maxWidth`-КЭПОМ (`width:'100%', maxWidth:200`), НЕ фиксированным `width`:** фикс-ширина —
  это пол, который на узкой колонке (мобильная полоса; лейбл-колонка FormRow фиксированная) выпирает за
  вьюпорт; кэп на широких выглядит так же, на узких сжимается (verified фиксером fanout-02: DatePicker
  `width:200` в 375-колонке давал +21px оверфлоу документа).
- **Ширина полосы формы ограничена** (`maxWidth` ≈ 560–720px), не во весь экран.
- **Описание поля — два места:** короткое важное для заполнения → под контролом (field-description);
  длинное / второстепенное / «почему так» → в попап-подсказку рядом с подписью, чтобы форма не разъезжалась
  вертикально.
- **Контрол под задачу:** точное число → number-input; диапазон / процент с наглядностью → slider;
  2–4 взаимоисключающих, видны сразу → сегменты; длинный текст → textarea.
- **Один размер контролов** на странице; иерархия spacing (между группами больше, внутри группы меньше).

## Контракт (feedback + data-safety → interface-foundations)

- **Submit — видимый pending / error** (interface-foundations §3): кнопка `disabled` + loader пока идёт; ошибка
  видима, форма НЕ закрывается.
- **Валидация — per-field:** подсветка невалидного поля + текст ошибки у поля (не только общий баннер).
- **Несохранённый ввод → защита** (interface-foundations §2): форма / диалог с введёнными данными не закрывается
  по outside-click / `Esc` без подтверждения. Пустую / нетронутую — можно.
- **Опасное действие** — через подтверждение, не прямой submit.

Интеракционные состояния (submit-pending / submit-error) тоже пресеть сценарием и задекларируй в
`scenarios.manifest.json` → `scaffold-app-shell` «Сценарии состояний» (для многошаговых флоу — и `step-2` / `done`).

**Экран результата флоу (done/success)** — это `PlaceholderContainer` с иллюстрацией (`SuccessOperation`,
вертикально: `direction="column" align="center"`), НЕ самодельный центр из большого `Icon` — механика и
покраска `--gil-color` → `library-illustrations`.

## Ручки сервиса (profile)

Точные числа ширины, политика «единая ширина vs по контенту», дефолт размещения описаний (под контролом vs
попап-подсказка), выравнивание / адаптивность страницы. Омиссия = разумный дефолт; финальное — профиль сервиса.
